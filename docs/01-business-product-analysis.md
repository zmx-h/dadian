# 搭电 (Dā Diàn) — 产品与业务分析

> 版本: v1.6 | 日期: 2026-06-21 | 目标形态: Web App (Desktop + Tablet)

---

## 1. 产品价值与定位

### 1.1 一句话定位
一个把出门变成 RPG 游戏的 AI 电子搭档——用游戏化思维解决年轻人「出门难」的全链路体验。

### 1.2 核心价值三角

| 维度 | 痛点 | 解决方案 | 壁垒 |
|------|------|---------|------|
| **决策** | 选择困难症，刷 3 小时定不了去哪儿 | 渐进式承诺（区域→点→路线）+ 灵感骰子兜底 | 拆解决策心理学应用到出行场景 |
| **社交** | 内向者惧怕社交消耗，多人局无人迁就 | 社交电量系统 + 最低电量基准 | 独特的社交建模 |
| **记忆** | 拍了照片相册吃灰 | VLM 打卡 + AIGC 回忆录 | 内容和情感资产沉淀 |

### 1.3 目标用户画像

**核心用户（18-32 岁城市青年）：**
- **i 人特种兵**（25%）：想把出门变成可控的冒险，需要工具来推自己一把
- **佛系 NPC**（20%）：不想动脑子，跟着走就好——**但这群人有流失风险**：如果 Onboarding 强制选角色，NPC 会在第一步被筛掉。解法：注册后不强制选角色，搭电根据三问结果推荐一个角色，用户可以直接跳过出发
- **社交续航焦虑者**（30%）：想出门但要精打细算社交能量
- **异地好友**（15%）：和朋友在不同城市，想一起玩
- **记忆收藏家**（10%）：痴迷记录生活，但讨厌传统的打卡方式

### 1.4 商业模式建议

| 层级 | 策略 | 说明 |
|------|------|------|
| **完全免费** | 全功能开放，不做付费墙 | MVP 阶段验证产品价值，不设功能限制 |
| **打赏入口** | 回忆录生成页底部放一个低调的打赏按钮 | 「觉得回忆录写得不错？请搭电喝杯咖啡 ☕」 |
| **远期 B 端（P2）** | 商圈/街区定制剧本、品牌惊喜事件 | 如「淮海路 × 搭电」专属出行剧本 |

## 2. 产品架构重设计

### 2.1 问题诊断：原设计缺陷

| 问题 | 影响 | 修正方向 |
|------|------|---------|
| 纯移动端思维 | 桌面端体验未真正设计 | 重新设计 Web 版三层布局 |
| 社区板块无详细设计 | 社交闭环缺失 | 补充社区 spec |
| POI 数据源直接依赖外部平台 | 成本高、延迟大、法律风险 | 自建 POI 中间层 + 缓存 |
| AI 成本未评估 | 规模化后成本爆炸 | 分层 AI 策略 |
| 离线能力未考虑 | 地下/信号差场景无体验 | 本地缓存 + 乐观更新 |
| 安全模型缺失 | 位置隐私、社交数据泄露 | 隐私设计文档 |

### 2.2 四板块关系（修正版）

```
出发（准备仪表舱）
  ↓ 按 [🚀 出发]
探索（实时旅程核心）
  ↓ 行程结束
回忆（内容资产产出）
  ↓ 选择分享
社区（社交分发与消费）
  ↕ 发现灵感 → 回到出发
```

### 2.3 功能优先级矩阵（MVP → V2）

| P0 MVP | P1 迭代 | P2 远期 |
|---------|---------|---------|
| 单人出行闭环 | 多人组队（同城） | 远程联机 |
| 社交电量 | 命运骰子 | 人格成长引擎 |
| 基础路线规划 | VLM 打卡 | AIGC 回忆录高级风格 |
| 锚定任务 | 彩蛋任务 | 足迹共振 |
| 简单回忆总结 | 一键转职 | 城市玩家年鉴 |
| 基础 POI 浏览 | 盲盒事件 | 社区完整功能 |

## 3. 关键设计决策

### 3.1 Web-First 设计原则

原设计是移动端，Web 端需要保留沉浸感同时利用大屏优势：

- **保留沉浸感**：暗色主基调、模糊与透明玻璃材质、细微动效
- **利用大屏**：出发页采用中央驾驶舱 + 两侧信息面板；探索页全屏地图 + 侧边任务面板
- **不引入触屏专用交互**：去掉摇一摇、长按、双指缩放依赖；替换为 hover/click/右键菜单
- **导航适配**：Dock 在桌面端替换为顶部标签行 + 侧边栏（≥1024px），平板保留悬浮胶囊

### 3.2 第三方平台集成策略

高德已直接接入。大众点评/小红书没有公开 API，不可直接依赖。

**POI 数据策略：**
- **主数据源**：高德 POI 搜索 + 逆地理编码（可靠、有 API）
- **用户贡献层**：出行过程中采集的数据（足迹、评价、照片）| 自有存储，是长期壁垒
- **热数据**（当前城市热门 POI）：Redis，1h TTL
- **温数据**（历史 POI）：PostgreSQL + PostGIS
- **富文本/标签**：DeepSeek 离线批量生成 POI 的「搭电话术」描述文案

### 3.3 AI 成本分层策略

| 场景 | 模型级别 | 延迟要求 | 调用频率 |
|------|---------|---------|---------|
| VLM 视觉打卡评价 | DeepSeek-VL | <1s | 每次出行 3-10 次 |
| 回忆录生成 | DeepSeek-Chat | <30s | 每次出行 1 次 |
| 人格推理更新 | 本地规则引擎 | 异步 | 每 5 次出行 1 次 |
| POI 标签抽取 | 批量离线 | 天级 | 定时任务 |

**DeepSeek 成本估算（按 500 DAU，日活 20% 出行率）：**
- 100 次出行/天 × 5 次 VLM = 500 次 VLM/天
- 100 次出行/天 × 1 次文案 = 100 次文案/天
- 预估月消耗：~¥200-500（DeepSeek 定价约为 GPT-4o 的 1/20）

### 3.4 实时通信策略

| 场景 | 技术选型 | 原因 |
|------|---------|------|
| 队友位置同步 | WebSocket（Socket.IO） | 低频更新（5s 间隔），双向 |
| 路线进度推送 | SSE | 服务端单向，断线重连友好 |
| 骰子软锁倒计时 | 客户端本地计时 + 服务端兜底 | 避免网络延迟误判 |
| 社区动态刷新 | Polling + SWR | 非实时场景，无需长连接 |
| 足迹共振通知 | WebSocket 或 Push | 需要即时性 |

## 4. 风险与建议

### 4.1 技术风险

| 风险 | 影响 | 缓解 |
|------|------|------|
| 高德/地图 API 限流或停服 | 路线规划不可用 | 自建 POI 数据缓存 + 降级路线估算 |
| VLM 调用成本失控 | 月账单超预算 | 设置硬额度上限 + 采样策略 |
| 用户生成内容（图片）审核 | 合规风险 | 接入内容安全 API + 先发后审（异步审核兜底） |

### 4.2 产品风险

| 风险 | 影响 | 缓解 |
|------|------|------|
| 社交电量概念太抽象 | 新用户不理解 | Onboarding 三问直接量化 + 给出参考锚点（"大多数和你选择一样的用户，电量在 XX-XX 之间"） |
| 命运骰子被误认为赌博 | 政策风险 | 设计为纯娱乐，无任何付费/实物兑换 |
| 用户活跃度「出门即流失」 | 留存低 | 回忆录是回访钩子；社区维持日常活跃 |
| **未成年人使用** | 严重合规风险 | 位置追踪 + 社交 + 照片上传 + 随机奖励组合天然吸引未成年人。注册流程增加「我已满 18 周岁」声明勾选框，隐私协议说明未成年人使用限制 |

### 4.3 合规风险

| 风险 | 影响 | 缓解 |
|------|------|------|
| **无注销/数据导出接口** | 《个保法》第 47 条违规 | 04 API 已补充 `DELETE /api/v1/users/me`（注销）、`POST /api/v1/users/me/export`（导出） |
| 社交电量量化无参考锚点 | 用户对数值不信任 | Onboarding 结果页展示「你的电量 30」+ 参考「大多数相似用户 30-45」或「Xxx 初始电量也是 30」 |

### 4.4 建议

1. **MVP 聚焦单人出行闭环**（出发→探索→回忆），社区和多人组队放 P1
2. **先做 Web 版架构设计时预留 API**，避免后续移动端重构
3. **AI 成本从 MVP 第 1 天就监控**，建立按用户维度的 token 成本仪表盘
4. **隐私设计前置**：用户位置数据默认不出行后 7 天匿名化处理
5. **社区板块缺 spec**：建议单独补一个完整设计文档（已补为 06-community-spec.md）

### 4.5 基础设施月度成本估算

MVP 阶段（500 DAU，100 出行/天）全栈 burn rate：

| 资源 | 规格 | 月估算 (¥) |
|------|------|-----------|
| ECS 2c4g | 常驻实例 | ~250 |
| RDS PostgreSQL | 2c4g 基础版 | ~150 |
| Redis 7 | 256MB 基础版 | ~80 |
| OSS + CDN | 标准存储 + 流量 | ~80 |
| OpenSearch | 最简配置 | ~150 |
| 阿里云短信 | 500 DAU × 10% 日登录 | ~68 |
| DeepSeek API | 100 出行/天 | ~350 |
| **合计** | | **~1,128/月** |

- 此 burn rate 在 MVP 阶段可控
- 免费+打赏模式下，收支平衡需要约 190 人/月打赏 ¥6（约 38% 的日活打赏率，现实中很难达到）
- 如果 DAU 增长到 5,000，主要增量来自 DeepSeek API（~¥3,500/月）和 ECS 升级（~¥500/月），合计约 ¥5,000/月
- 建议上线后第一个月即开始追踪基础设施成本表，确认实际与估算的偏差
